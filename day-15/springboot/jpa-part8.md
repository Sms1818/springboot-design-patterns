# 🔥 What are Derived Queries?

---

## 📌 Definition

Derived Queries are queries automatically generated by Spring Data JPA from method names.

---

## 🧠 Core Idea

```text
Method Name → Parsed by JPA → SQL Query Generated
```

---

# 🔥 Why Derived Queries?

---

## ✅ Advantages

- No manual SQL required
- Faster development
- Readable code
- Database independent
- Automatically translated into SQL

---

# 🔥 Important Rule

---

## 📌 Naming Convention Matters

JPA internally parses method names using:

```text
PartTree.java
```

If naming is incorrect:

```text
Query generation fails
```

# 🔥 JPQL (VERY IMPORTANT)

---

# 🔥 What is JPQL?

---

## 📌 Definition

```text
Java Persistence Query Language
```

---

## 🧠 Important

JPQL works with:

```text
Entity names + Entity fields
```

NOT:

```text
Table names + Column names
```

---

# 🔥 Why JPQL?

---

## ✅ Advantages

- Database independent
- Object-oriented query language
- Supports joins
- Easier entity mapping

---

# 🔥 Basic JPQL Query

---

## 💻 Repository

```java
@Query("""
SELECT u
FROM UserDetails u
WHERE u.name = :userFirstName
""")
List<UserDetails> findByUserName(
        @Param("userFirstName")
        String userName
);
```

---

# 🔥 Understanding Query

---

```text
u → alias
UserDetails → entity name
u.name → entity field
:userFirstName → named parameter
```

---

# 🔥 Important Rule

---

## ⚠️ Return Type Must Match Result

If query returns multiple rows:

```java
List<UserDetails>
```

If you use:

```java
UserDetails
```

👉 Exception occurs

# 🔥 N+1 Problem (VERY IMPORTANT)

---

# 🔥 What is N+1 Problem?

---

## 📌 Scenario

```text
1 User → Many Addresses
Query fetches multiple users
```

---

# 🔥 What Happens?

---

## 🧠 Flow

```text
1 Query → fetch all users

For each user:
1 Query → fetch addresses

Total:
N + 1 Queries
```

---

# 🔥 Example

---

```text
100 users

1 query → users
100 queries → addresses

Total = 101 queries
```

---

# 🔥 Important

---

## ❌ EAGER does NOT solve it

Because:

```text
Multiple Parent + Multiple Child
still causes multiple queries
```

---

# 🔥 Solution 1 → JOIN FETCH (BEST)

---

## 💻 Query

```java
@Query("""
SELECT ud
FROM UserDetails ud
JOIN FETCH ud.userAddressList ad
WHERE ud.name = :userFirstName
""")
List<UserDetails> findUserDetailsWithAddress(
        @Param("userFirstName")
        String userName
);
```

---

# 🔥 Why JOIN FETCH?

---

## 🧠 Behavior

```text
Fetch parent + child
in SINGLE query
```

---

# 🔥 Solution 2 → @BatchSize

---

## 💻 Entity

```java
@OneToMany(
        cascade = CascadeType.ALL,
        fetch = FetchType.EAGER
)
@BatchSize(size = 10)
@JoinColumn(name = "user_id")
private List<UserAddress> userAddressList;
```

---

# 🔥 Behavior

---

## 🧠 Important

```text
Does NOT make single query
BUT reduces total queries
```

# 🔥 Flush and Clear (VERY IMPORTANT)

---

# 🔥 Problem

---

## 🧠 Persistence Context Still Holds Old Data

---

## 💻 Service

```java
@Transactional
public void deleteByUserName(String name) {

    userDetailsRepository.findById(1L).get();

    userDetailsRepository.deleteByUserName(name);

    Optional<UserDetails> output =
            userDetailsRepository.findById(1L);

    System.out.println(
            "output present: " +
                    output.isPresent()
    );
}
```

---

# 🔥 Why Output Still Present?

---

Because:

```text
Persistence Context cache still contains entity
```

---

# 🔥 Solution

---

## 💻 Repository

```java
@Modifying(
        flushAutomatically = true,
        clearAutomatically = true
)
@Query("""
DELETE FROM UserDetails ud
WHERE ud.name = :userFirstName
""")
void deleteByUserName(
        @Param("userFirstName")
        String userName
);
```

---

# 🔥 What Happens?

---

## 🧠 flushAutomatically

```text
Push changes to DB immediately
```

---

## 🧠 clearAutomatically

```text
Clear persistence context cache
```

# 🎯 Interview Questions

---

## 🔹 What are Derived Queries?

Auto-generated queries from method names.

---

## 🔹 Does Derived Query support INSERT?

❌ No

Use:

```java
save()
```

---

## 🔹 Difference:
JPQL vs SQL?

| JPQL | SQL |
|---|---|
| Entity based | Table based |
| DB independent | DB dependent |
| Uses fields | Uses columns |

---

## 🔹 What is N+1 problem?

Multiple extra child queries for parent records.

---

## 🔹 Best solution for N+1?

```text
JOIN FETCH
```

---

## 🔹 Why @Modifying needed?

To allow UPDATE/DELETE/INSERT queries.

---

## 🔹 Why flushAutomatically?

Push changes immediately to DB.

---

## 🔹 Why clearAutomatically?

Clear stale persistence context cache.

---

## 🔹 Why use DTO projection?

Avoid Object[] handling.

---

# 🚀 Final Summary

---

```text
Derived Queries:
- Auto-generated
- Naming convention based

Pagination:
- Pageable
- Page
- Sort

JPQL:
- Entity based query language
- Supports joins
- Database independent

N+1 Problem:
- Multiple extra queries
- Solved using JOIN FETCH

@Modifying:
- Required for UPDATE/DELETE

Flush:
- Sync persistence context → DB

Clear:
- Remove cached entities
```
